# Requirements: カラム単位リモート操作（column-remote-control）

## 概要

### Why
editor-spotlighter のリモートビューは、スマホからの画面閲覧と入力送信を提供しているが、テキスト入力は常に「Claude Code 全体」へ送られる（src/extension.ts:691-731 の type 受信処理は、クリップボード書込 → `claude-vscode.focus` → 前面アプリ検査 → Cmd+V+Enter の順で、列＝エディタグループを一切選ばない）。一方、スマホ側の列選択（selectColumn）は閲覧画面のクロップ切替のみに使われ、入力先に連動しない（src/remote/remoteViewServer.ts:464-472）。このため、複数カラムで複数の Claude Code セッションを並列運用していると、「いま見ている列のエージェント」に指示を出したつもりでも、実際にはフォーカス状態次第で別セッションに届く。

### 対象ユーザー
外出先からスマホで自宅/オフィスの VS Code を遠隔監視・操作する開発者（本拡張のオーナー1名運用が前提。ただし同一人物が複数端末から接続する場面はある）。

### 行動変容（1行）
外出先のスマホから「いま画面に表示している列のエージェントだけ」に確実に指示を送れるようになり、複数エージェント並列運用を遠隔から安心して回せる。

## ユーザーストーリー

### 主シナリオ: 外出先から特定列のエージェントに指示を出す
開発者として、外出先のスマホでリモートビューに接続し、列2で動いている Claude Code セッションの作業状況を確認したうえで、列2のセッションにだけ追加指示を送りたい。なぜなら、列1・列3では別タスクのエージェントが並走しており、指示が別の列に届くと無関係なタスクが汚染されるから。

- スマホで列2を選択すると、画面は列2の単列表示に切り替わり、入力欄にも「送信先が列2である」ことが表示される。
- テキストを送信すると、PC側で列2のエディタグループへのフォーカス確定が検証されてから貼り付け・実行され、列2のセッションだけが指示を受け取る。
- 送信後のスクリーンキャプチャ更新で、列2に指示が入力されたことをスマホ画面上で確認できる。

### 副シナリオ1: 複数列を巡回して状況確認だけする（閲覧専用）
開発者として、移動中に列1→列2→列3と順に切り替えて各エージェントの進捗を眺めたい。入力はしない。列切替は1タップで完結し、切替のたびに最新のキャプチャが表示される（既存挙動の維持）。

### 副シナリオ2: 全列を俯瞰してから対象列に入る
開発者として、どの列のエージェントが停止・質問待ちになっているかを一目で把握してから、その列に入って指示したい。列選択UIには各列の識別情報（その列のアクティブタブ名）が表示され、当て推量の番号タップではなく内容で列を選べる。

### 副シナリオ3: 安全な待ち受け設定への移行
開発者として、リモートビューサーバの待ち受けが必要時以外に LAN へ露出しない状態にしたい。ただし、これまで使っていた「同一 LAN 内のスマホで QR コードを読んで直結する」運用は引き続き選択でき、移行によって「QR を読んでも繋がらない理由が分からない」状態に陥らないこと。

## サービスフロー

### 時系列フロー（主シナリオの一連）

```
スマホ（フロントステージ）          PC側（バックステージ）
─────────────────────────────────────────────────────────
1. QR読取/URL入力 → ログイン画面    パスワード検証・セッション発行
2. 接続完了                         WS確立・初回キャプチャ・列数/タブ情報の送信
3. 列バーで列2をタップ              selectColumn受信 → 選択列を更新 →
                                    列2クロップのフレーム配信（単列表示）
4. 入力欄に「送信先: 列2」表示      （状態表示のみ・PC操作なし）
5. テキスト入力 → 送信              type受信 → 検証 →
                                    ①列2のエディタグループへフォーカス
                                    ②フォーカス確定の検証（確定するまで貼り付けない）
                                    ③前面アプリ＝VS Code の検査
                                    ④貼り付け＋Enter
                                    ⑤失敗時: 注入中止＋スマホへ結果通知
6. 結果確認                         送信直後にキャプチャ更新 → 列2の最新画面配信
7. 切断（×ボタン）                  キャプチャ停止・レイアウト復元
```

### 状態遷移

- **未接続**: ログイン前。画面・入力とも不可。
- **接続済み・列選択中**: 既定では前回選択列（初回は列0）。閲覧は常時可。入力欄は選択列を送信先として表示。
- **入力送信中**: 送信〜PC側の注入完了/中止まで。この間スマホ側は送信結果（成功/中止＋理由）の通知を受け取れる。連打による多重注入は抑止される。
- **入力不可（閲覧専用）**: PC側設定でリモート入力が OFF のとき。閲覧と列切替は可、入力欄は不可状態を明示。
- **切断/再接続中**: WS断。再接続バナー表示（既存挙動）。再接続後は列選択状態をサーバから受信した最新値に同期する。

### 前後フローとの接続
- 上流: PC側でリモートビューサーバを起動し、サイドバーに QR・URL・パスワードを表示する既存フロー（src/extension.ts の起動処理）。本機能は待ち受けアドレスの既定値変更（要件c）でこの起動フローの表示内容に影響する。
- 下流: 注入されたテキストは各列の Claude Code セッションが処理する。本機能のスコープは「正しい列に届くこと」までで、セッション側の応答内容は対象外。

## 機能要件

### (a) 指定列への入力ルーティング

- **a-1. 入力先は「スマホで選択中の列」に一致させる**: type メッセージ受信時、現在の選択列に対応するエディタグループ内の Claude Code 入力欄へフォーカスを移してから貼り付け・実行する。現行の「常に claude-vscode.focus（全体フォーカス）」は選択列に基づくフォーカスに置き換える。
  - 受け入れ条件: 3列以上で各列に別セッションが動いている状態で、スマホから列を切り替えながら送信した各テキストが、それぞれ選択していた列のセッションにのみ入力される。
- **a-2. 順序保証（フォーカス確定 → 貼り付け）**: 貼り付け・Enter の注入は、対象列へのフォーカスが確定したことを検証してから実行する。固定待ち時間（現行の setTimeout 500ms）だけを根拠に注入しない。フォーカス確定が検証できない場合は注入を中止する（fail-closed）。
  - 受け入れ条件: フォーカス確定検証が成功した場合のみ注入が実行され、検証失敗時は注入ゼロ＋ユーザー通知になることがテストで確認できる。
- **a-3. 注入結果のフィードバック**: 注入の成功/中止（中止理由: フォーカス未確定・前面アプリ非VS Code・入力許可OFF等）をスマホ側に通知し、ユーザーが「届いたか分からないまま再送する」状況をなくす。
  - 受け入れ条件: 注入中止が発生したとき、スマホ画面に中止と理由が表示される。
- **a-4. 前面アプリ検査の維持**: 既存の「前面アプリが VS Code でなければ注入しない」ガード（src/extension.ts:709-731）は本改修後も維持する。
  - 受け入れ条件: 前面に別アプリがある状態での送信が注入ゼロで中止される（既存挙動の回帰なし）。

### (b) モバイルUIの列選択・単列拡大表示の体験設計

- **b-1. 閲覧列＝入力先列の同一性表示**: 画面に表示中の列と入力の送信先列が常に同一であることを、入力欄付近の表示（例: 「→ 列2」のような送信先表示）でユーザーに保証する。表示と送信先が食い違う状態を UI 上に存在させない。
  - 受け入れ条件: 列切替の直後から、入力欄の送信先表示が選択列と一致している。
- **b-2. 列の識別情報表示**: 列選択UIは番号だけでなく、各列のアクティブタブ名（既存の tabs 情報を利用）を併記し、内容で列を選べるようにする。
  - 受け入れ条件: 列選択UIに各列のアクティブタブ名が表示され、タブ切替時に追従する。
- **b-3. 列数の動的追従**: 列選択UIは固定4ボタン（src/remote/mobileHtml.ts:181-186）ではなく、サーバから受信した列数（columns メッセージ）に応じて生成・更新する。
  - 受け入れ条件: 列数2・5の構成それぞれで、列選択UIのボタン数が実列数と一致する。
- **b-4. 単列拡大表示の維持と切替応答**: 選択列のクロップ表示（既存実装）を維持し、列切替時は新しい列のフレームが届くまで「切替中」であることが分かる表示にする（古い列の画像を新しい列と誤認させない）。
  - 受け入れ条件: 列切替操作後、新フレーム受信までの間に旧列の画像が「選択列の現在画面」として誤認される表示にならない。

### (c) 待ち受けアドレスの 127.0.0.1 既定化（体験面の要件）

- **c-1. 既定値の変更**: 待ち受けアドレス設定（現行既定 "0.0.0.0"、src/extension.ts:665）の既定を 127.0.0.1 に変更する。Cloudflare Tunnel 等のトンネル経由運用は cloudflared が localhost から接続するため既定値のまま動作する。
  - 受け入れ条件: 設定未変更の新規環境で、サーバが 127.0.0.1 のみで待ち受ける。
- **c-2. LAN直結QR運用との両立**: LAN 直結（同一 LAN のスマホから QR で接続）を使う場合は、設定で LAN 待ち受けを明示的に有効化できる。サイドバーの QR・URL 表示は待ち受け設定と整合させる: 127.0.0.1 待ち受けかつトンネル未設定のときは LAN IP の URL/QR（繋がらない QR）を表示せず、「LAN 直結を使うには設定変更が必要」であることと設定への導線を表示する。
  - 受け入れ条件: 127.0.0.1 待ち受け・トンネル未設定の状態で、接続不能な LAN URL の QR が表示されない。
- **c-3. 移行時の混乱防止**: 既定値変更後の初回起動時、これまで LAN 直結を使っていたユーザーが「QR を読んでも繋がらない」に陥らないよう、待ち受けが localhost のみであることと LAN 直結への切替方法を起動時メッセージまたはサイドバーで1回明示する。
  - 受け入れ条件: 既定値変更後の初回起動で、待ち受け範囲の変更と切替手順がユーザーに提示される。

## エッジケース

| # | ケース | 要求挙動 |
|---|--------|---------|
| 1 | **列構成が手元とスマホ表示でズレた時**（画像解析の列境界検出と実際のエディタグループ数が不一致。src/remote/panelDetector.ts はセパレータ未検出時に1カラム扱いへフォールバックする） | 入力ルーティングの列対応は画像解析結果ではなく VS Code API のエディタグループ（tabGroups）を真実源とする。選択列番号が実グループ数の範囲外なら注入を中止しスマホへ通知する。表示クロップのズレは閲覧品質の問題に留め、入力先の誤りには波及させない。 |
| 2 | **フォーカス確定失敗時**（対象列へのフォーカスコマンド失敗・確定検証タイムアウト） | 注入を一切行わず中止し、スマホへ中止理由を通知する。リトライは自動で行わずユーザーの再送に委ねる（自動リトライはタイミングずれによる誤爆経路になるため）。 |
| 3 | **入力許可OFF時**（リモート入力許可設定が false） | 既存どおりサーバ側で type/click/switchTab を破棄する。加えてスマホUIで入力欄を不可状態にし「閲覧専用」であることを表示する（送信したのに無反応、を作らない）。 |
| 4 | **複数クライアント同時接続時**（選択列はサーバ側の単一共有状態。src/remote/remoteViewServer.ts:36） | 全クライアントが同一の選択列を共有する仕様と定める（1人運用前提）。クライアントAの列切替は columns メッセージで即時に全クライアントへ同期され、クライアントBの画面表示・送信先表示も追従する。「Bの画面表示は旧列のまま、送信はAが切り替えた列に届く」という不一致状態を残さない。 |
| 5 | **リモート操作中に列の増減が起きた時**（PC側でエディタグループが追加・削除される） | 列数変更は即時に全クライアントへ配信し、列選択UIを更新する。選択中の列が消滅した場合は選択を有効範囲内（例: 末尾列）へ寄せ、送信先表示を更新したうえで、変更後の最初の送信前にユーザーが送信先を視認できる状態にする。列削除と送信が競合した場合（送信処理中にグループが消えた）は注入を中止し通知する。 |
| 6 | **再接続直後**（WS切断→自動再接続） | 再接続時にサーバの最新状態（列数・選択列・タブ情報）を受信してから入力可能にし、切断前のローカル状態を信用しない。 |

## 非機能要件

- **誤爆防止（最重要）**: 意図しない列・意図しないアプリへの入力注入をゼロにする。判断原則は fail-closed — 「対象列へのフォーカス確定」「前面アプリ＝VS Code」「選択列が有効範囲内」「入力許可ON」のいずれか1つでも検証できなければ注入しない。検証不能時に「たぶん大丈夫」で注入する経路を仕様上存在させない。
- **応答性**: 列切替からその列の最新フレーム表示までは、既存の selectColumn → captureOnce と同等の体感を維持する。入力送信からフォーカス確定検証・注入完了までに追加する検証処理は、現行の固定待ち（500ms）に対して体感を悪化させない範囲とする（厳密な数値目標より誤爆ゼロを優先し、トレードオフが生じたら安全側に倒す）。
- **既存閲覧機能の互換**: 画面閲覧・タップ操作（click）・タブ切替（switchTab）・再接続・明示切断・セッション失効管理・ログインロックアウトの既存挙動を変えない。
- **セキュリティ既存ガードの維持**: 入力長上限・型検証・セッション有効性の毎メッセージ再確認・接続元IP許可判定（src/remote/remoteViewServer.ts:516-544）等の既存防御は本改修で弱めない。待ち受け既定の 127.0.0.1 化はこの防御の強化側変更である。

## スコープ外

- macOS 以外の OS 対応（screencapture / osascript / swift 依存の現行構造は維持）。
- 複数 VS Code ウィンドウの同時リモート対応（対象は単一ウィンドウ）。
- 複数クライアントごとに独立した列ビュー・独立した入力先を持たせる機能（選択列は全クライアント共有のまま）。
- Claude Code 以外のターゲット（任意のエディタ入力欄・ターミナル等）への汎用入力ルーティング。
- リモートビューの残存セキュリティ債務の解消（Cookie の常時 Secure 化・/tmp スクリーンショットの保存先見直し・サーバ自動テスト整備等）。これらは別タスクとして扱う（c の bind 既定化のみ本スコープ）。
- 画面ストリーミングの方式変更（キャプチャ間隔・画質・転送方式の改善）。
- スマホUIの全面リデザイン（本件は列選択・送信先表示・列数追従に必要な範囲の変更に限る）。
